Local Security Policy

For Windows and Windows Server

Managing the local security policies on a system is very important. They control things ranging from password requirements to niche, technical security options. Improperly configured security policies pose a risk to the security of the system that can easily be remedied.

Account Policies

The policies in the Account Policies branch of secpol relate to password requirements and locking accounts after failed sign-ins.

Password Policies

Included here are the policies defining password requirements. In order, these are:

  • Enforce Password History

    • Recommended: 5
    This is the number of previous passwords to save, in order to prevent users from rotating between a small number of passwords when asked to change them.

  • Maximum Password Age

    • Recommended: 90
    This number, in days, is how long a user can keep the same password before being forced to change it. These prevents old, potentially compromised passwords from remaining on the system.

  • Minimum Password Age

    • Recommended: 30
    This number, in days, is how long after changing a password before a user can change it again.

  • Minimum Password Length

    • Recommended: 14
    This is the minimum number of characters that a valid password can be. 14, for historical reasons, is the maximum on windows machines. It's also a good length for us.

  • Password Must Meet Complexity Requirements

    • Recommended: Enabled
    Without this enabled, none of the other password requirements take effect. They're not much use to us unless they're turned on, so make sure to enable this.

  • Store Passwords Using Reversible Encryption

    • Recommended: Disabled
    Windows already stores the passwords encrypted by default. This makes that encryption reversible, meaning that passwords can be learned just by finding them and unencrypting them. Usually, they're stored using unreversible encryption, so enabling this is a security risk.

Account Lockout Policy

These three policies dictate what should happen after consecutive failed login attempts:

  • Account Lockout Duration

    • Recommended: 30
    This number, in minutes, is how long the account should be locked out once enough failed logins occur.

  • Account Lockout Threshold

    • Recommended: 5
    This is how many failed attempts are needed to trigger the aaccount lockout. It shouldn't be too low, otherwise a series of honest typos could trigger it. It also shouldn't be too high, or it'll give malicious actors more attempts to crack the password. 5 is a good middle-ground.

  • Reset Account Lockout Counter After

    • Recommended: 30
    This is how many minutes it takes for the failed login attempt counter to reset.

Local Policies

The Local Policies branch is much more varied, containing the Audit Policy, User Rights Assignment, and Security Options nodes.

Audit Policy

The Audit Policy is by far the simplest of the three. All you need to do is go into every option and check success and failure.

These options decide whether the computer keeps a log of administrative and other actions performed on it. How on earth you read them is beyond me, though.
